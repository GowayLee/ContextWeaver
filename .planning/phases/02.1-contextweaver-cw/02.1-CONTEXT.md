# Phase 02.1: contextweaver-cw - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

修复 `contextweaver` / `cw` 的 CLI 入口链路失声问题，确保用户直接运行入口命令时不会出现“没有任何输出”的静默失败。范围包含无参数启动、帮助触发，以及代表性子命令如 `index`、`search` 的入口派发与可见输出恢复；不扩展新命令，也不改变既有索引/检索业务语义。

</domain>

<decisions>
## Implementation Decisions

### CLI Entry Contract

- **D-01:** 直接运行 `contextweaver` 或 `cw` 且不带参数时，必须显示完整标准 help，而不是静默退出。
- **D-02:** 无参数显示 help 视为正常帮助展示，退出码固定为 `0`。
- **D-03:** 帮助触发行为要统一覆盖无参数、`help`、`-h`、`--help`，且 `contextweaver` 与 `cw` 两个别名行为一致。

### Repair Scope

- **D-04:** Phase 02.1 修复范围是整个 CLI 入口链路，不只修裸跑 help；`index`、`search` 等代表性子命令也必须恢复为“会产生符合预期的可见输出”。
- **D-05:** 优先定位并修复共享入口、bin 映射、构建产物或命令派发层的问题，避免分别在单个子命令上做零散补丁。

### Regression Protection

- **D-06:** 验收与回归保护至少覆盖这些用户入口：无参数、`help`、`-h`、`--help`、`index`、`search`。
- **D-07:** 回归检查必须同时覆盖 `contextweaver`、`cw` 两个别名，以及构建后的 `dist/index.js`，防止源码修复后发布入口再次静默失效。

### the agent's Discretion

- help 的具体排版与 `cac` API 选型由实现阶段决定，但必须满足上面的输出与退出码契约。
- 测试可采用子进程 smoke test、入口层单测或两者组合，由实现阶段根据最小可靠改动决定。

</decisions>

<code_context>

## Existing Code Insights

### Reusable Assets

- `src/index.ts`：唯一 CLI 入口，集中做 `cac('contextweaver')` 初始化、命令注册、help 注册与 `cli.parse()` 调用。
- `package.json`：`contextweaver` 与 `cw` 都映射到同一个 bin `dist/index.js`，适合从共享入口层一次修复两个别名。
- `tests/indexCli.test.ts`：已有 CLI 相关测试基座与 `runIndexCliCommand` 测试模式，可扩展为入口/输出回归保护。

### Established Patterns

- CLI 的用户可见输出集中在边界层处理：`src/index.ts` 使用 `logger`、`process.stdout.write` 和 `process.exit()` 控制终端行为。
- 仓库当前倾向把命令编排保留在 `src/index.ts`，具体业务逻辑下沉到其他模块；入口问题应优先在入口边界修复。
- 现有测试更偏向命令 helper 与索引成败输出，尚未覆盖“直接运行 CLI 入口时的默认表现”这类 smoke 场景。

### Integration Points

- `src/index.ts`：help 展示、主模块判断、`cli.parse()` 调用顺序，是本 phase 最关键的集成点。
- `dist/index.js`：发布产物与全局安装实际执行的入口，需要纳入验证，避免仅源码正确。
- `README.md`：如果最终 help/入口契约需要更明确的用户提示，文档可能要与实现保持一致。

</code_context>
